**Critical implementation note:** These tests call repository/query functions directly but must pass the test `db` instance. Two approaches:
- Option A: Repository functions accept `db` parameter (dependency injection) — requires checking if repos accept db param or import it at module level
- Option B: If repos import db at module level, use `jest.mock` to override the db import with the test db — but this is an integration test, so prefer injection
- Option C: Call Drizzle directly in tests (db.insert, db.select, db.delete) to test the SQL queries that repositories wrap — most isolated approach

Read each repository file to determine its db import pattern. If repos use module-level `import { db } from '../database/client'`, then in integration tests either:
- Mock the `../database/client` module to return the test db, OR
- Use Drizzle operations directly (preferred for true integration testing)

Use Drizzle operations directly against the test db for maximum isolation. Test the same SQL patterns the repositories use, but with the real better-sqlite3 backend. This validates schema + queries without coupling to the repository module system.
